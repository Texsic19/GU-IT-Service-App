# Gonzaga University IT Help Desk Ticketing System

A full-stack IT ticketing system built for Gonzaga University's Information Technology Services department. Students, faculty, and staff submit support tickets through a clean web interface. IT technicians manage, assign, and resolve tickets with the help of AI-generated fix suggestions powered by Google Gemini.

## System Description

This system is an IT help desk ticketing platform for Gonzaga University. Students, staff, and faculty log in to submit IT support tickets describing their issue — the AI automatically categorizes the ticket (Network, Hardware, Software, etc.) and assigns a priority level (Low, Medium, High, Critical). IT technicians log in with a staff password to access the full dashboard, where they can view all tickets, assign them to technicians, update statuses, generate AI-suggested step-by-step fix instructions (similar to how Nessus provides remediation guidance for vulnerabilities), and apply tags. The system tracks which tickets are open, in progress, resolved, or closed, and maintains a full history of all ticket activity.

## Database Schema

### Table: `users`
Stores students, staff, and faculty who submit tickets.

| Column | Type | Constraints |

| id | SERIAL | PRIMARY KEY |
| first_name | VARCHAR(100) | NOT NULL |
| last_name | VARCHAR(100) | NOT NULL |
| email | VARCHAR(150) | UNIQUE, NOT NULL |
| phone | VARCHAR(20) | |
| department | VARCHAR(100) | |

### Table: `technicians`
Stores IT desk workers who resolve tickets.

| Column | Type | Constraints |
|--------|------|-------------|
| id | SERIAL | PRIMARY KEY |
| first_name | VARCHAR(100) | NOT NULL |
| last_name | VARCHAR(100) | NOT NULL |
| email | VARCHAR(150) | UNIQUE, NOT NULL |
| phone | VARCHAR(20) | |
| specialization | VARCHAR(100) | |
| is_active | BOOLEAN | DEFAULT TRUE |

### Table: `tickets`
Core ticket records — the main table of the system.

| Column | Type | Constraints |
|--------|------|-------------|
| id | SERIAL | PRIMARY KEY |
| title | VARCHAR(255) | NOT NULL |
| description | TEXT | NOT NULL |
| status | VARCHAR(50) | CHECK (Open/In Progress/Resolved/Closed), DEFAULT 'Open' |
| priority | VARCHAR(20) | CHECK (Low/Medium/High/Critical), DEFAULT 'Medium' |
| category | VARCHAR(100) | CHECK (Network/Hardware/Software/Account/Access/AV Equipment/Email/Printing/General) |
| submitter_name | VARCHAR(200) | Free-text name from submit form |
| submitter_email | VARCHAR(150) | Free-text email from submit form |
| submitter_id | INTEGER | FK → users.id ON DELETE SET NULL |
| assigned_tech_id | INTEGER | FK → technicians.id ON DELETE SET NULL |
| ai_suggested_fix | TEXT | Generated by Google Gemini |
| ai_categorized | BOOLEAN | DEFAULT FALSE |
| location | VARCHAR(150) | Campus location of the issue |
| created_at | TIMESTAMP | DEFAULT NOW() |
| updated_at | TIMESTAMP | DEFAULT NOW() |
| resolved_at | TIMESTAMP | Set when status changes to Resolved |

### Table: `tags`
Reusable labels that can be applied to tickets.

| Column | Type | Constraints |
|--------|------|-------------|
| id | SERIAL | PRIMARY KEY |
| name | VARCHAR(50) | UNIQUE, NOT NULL |
| color | VARCHAR(7) | Hex color code, DEFAULT '#3B82F6' |

### Table: `ticket_tags` *(junction table)*
Implements the many-to-many relationship between tickets and tags.

| Column | Type | Constraints |
|--------|------|-------------|
| ticket_id | INTEGER | FK → tickets.id ON DELETE CASCADE |
| tag_id | INTEGER | FK → tags.id ON DELETE CASCADE |
| | | PRIMARY KEY (ticket_id, tag_id) |

## Relationships

- One **user** submits many **tickets** (one-to-many): a student can submit multiple tickets over time.
- One **technician** is assigned many **tickets** (one-to-many): an IT worker can be responsible for multiple open tickets.
- One **ticket** can have many **tags**, and one **tag** can apply to many **tickets** (many-to-many): implemented via the `ticket_tags` junction table. For example, a ticket about a slow printer might be tagged with both "printer" and "wifi".

## Page-by-Page Plan

### Login (`pages/login.py`)
Users select their role (Student or IT Technician). Students click Continue with no password. IT staff enter a password. On success, `st.session_state` is set and `st.navigation()` renders only the appropriate pages for that role.

### Dashboard (`pages/dashboard.py`) — IT Staff only
Shows four live metric cards: Open Tickets, In Progress, Resolved (last 7 days), Critical Open. Two bar charts: tickets by category and tickets by priority. A recent tickets table showing the last 15 submissions with color-coded priority rows.

### Submit Ticket (`pages/submit_ticket.py`) — All logged-in users
Students type their full name, GU email, issue title, location, and description. They can optionally override category and priority; otherwise the AI auto-detects both via Google Gemini. On submit, the ticket is inserted into the database and a success confirmation is shown.

### Manage Tickets (`pages/manage_tickets.py`) — IT Staff only
Search and filter panel (by keyword, status, priority, category). Results shown as expandable rows. Each row lets staff update status, assign a technician (pulled from the `technicians` table), save changes, or delete with confirmation. A "View Full Ticket" button navigates to the detail page.

### Ticket Detail (`pages/ticket_detail.py`) — IT Staff only
Full view of a single ticket: description, submitter info, assigned technician, tags. AI-Suggested Fix panel powered by Google Gemini — generates step-by-step resolution guidance on demand. Quick Update form to change status and assignment. Tag management with a dynamic dropdown pulled from the `tags` table.

### Manage Users (`pages/manage_users.py`) — IT Staff only
Add new users via form with validation. Search users by name, email, or department. Edit existing users (pre-filled form). Delete with confirmation step.

### Manage Technicians (`pages/manage_technicians.py`) — IT Staff only
Add, edit, search, and delete technicians. Shows open and total ticket counts per technician. Active/inactive toggle.

### Tags (`pages/tags.py`) — IT Staff only
Add tags with a name and color picker. View all tags with usage counts. Delete tags with confirmation.

## Validation Rules

### Submit Ticket form
- Full name: required, cannot be blank
- GU email: required, must match regex `[^@]+@[^@]+\.[^@]+`
- Issue title: required, cannot be blank
- Description: required, minimum 20 characters

### Add/Edit User form
- First name: required
- Last name: required
- Email: required, must match email regex
- Phone: optional, if provided must match 10-digit format `\d{3}[-.\s]?\d{3}[-.\s]?\d{4}`

### Add/Edit Technician form
- First name: required
- Last name: required
- Email: required, must match email regex
- Phone: optional, same 10-digit format check

### Add Tag form
- Name: required, maximum 50 characters, must be unique

## Setup Instructions

### 1. Clone the repo
```bash
git clone https://github.com/yourusername/gu-it-service-app.git
cd gu-it-service-app
```

### 2. Set up Supabase
1. Go to [supabase.com](https://supabase.com) and create a free project
2. In the SQL Editor, paste and run the contents of `schema.sql`
3. Copy your connection string from: **Settings → Database → Connection string → URI**

### 3. Get a Gemini API key (free)
1. Go to [aistudio.google.com](https://aistudio.google.com)
2. Click **Get API Key → Create API key**

### 4. Configure secrets
Create `.streamlit/secrets.toml` (never commit this file):
```toml
[database]
DB_URL = "postgresql://postgres.xxxx:password@aws-0-us-west-1.pooler.supabase.com:5432/postgres"

[gemini]
GEMINI_API_KEY = "AIza..."

[auth]
staff_password = "your-it-staff-password"
```

### 5. Install and run
```bash
pip install -r requirements.txt
streamlit run app.py
```

## Features

- **Role-based access** via `st.navigation()` — students see only Submit Ticket; IT staff see the full dashboard
- **AI auto-categorization** — Google Gemini classifies ticket category and priority on submission
- **AI fix suggestions** — Gemini generates Nessus-style step-by-step remediation guidance per ticket
- **Full CRUD** on tickets, users, technicians, and tags
- **Search and filter** on the Manage Tickets page
- **Many-to-many tagging** via `ticket_tags` junction table
- **Parameterized SQL** throughout — no string concatenation in any query
- **Secrets management** via `st.secrets` — no credentials in code or repo
